# InnerVerse Web App — Build Decisions

Decisions locked in for the build. Read this before starting any work in this project.

## Project Goal

Build a web app version of InnerVerse to unblock Android testers and serve as the foundation for the full product. Accessed via URL (no app store submission). The Bubble iOS app will eventually be replaced by this build as it matures.

This is **not** a throwaway prototype. It's the foundation that will grow progressively into the full product.

## Scope: v1 Functionality Tiers

The visual structure of the full app is built — it should look like the existing Bubble version. Functionality is tiered:

### Tier 1 — Fully functional (must work like Bubble)

**Coaching sessions are the highest priority.** They must match the current Bubble app's behavior in every meaningful way.

- Login / Sign-up (magic link via Clerk)
- **Onboarding flow** — all 6 steps work, selections save to database AND flow into the coaching prompt at runtime (coach name → ai_persona, coaching style → coaching_style, goals → goals, etc.)
- **Coaching session chat interface** — text input, message display, streaming responses, end session button
- **Coaching prompt assembly** — uses the existing pattern from Bubble: `/v1/responses` endpoint with prompt assembled from variables (user_name, coaching_style, ai_persona, style_calibration, recent_style_feedback, goals, recent_breakthroughs, last_session_summary)
- **Cross-session memory** — sessions remember context from previous conversations
- **Session-end processing** — equivalent of Bubble's `process-session-end-v3`:
  - Generate session summary (becomes last_session_summary for next session)
  - Create breakthroughs (via equivalent of create-breakthrough)
  - Create insights / mindset shifts (via equivalent of create-mindset-shift)
  - Create next steps (via equivalent of create-next-step)
  - Update goal progress (via equivalent of update-goal-progress)
- **Session reflection + feedback survey** after ending a session

### Tier 2 — Visually present, minimal functionality

These tabs exist in the navigation and look like the current app, but interactivity is limited in v1:

- **Home tab** — shows "Start a new session" entry point and last session summary (read from database)
- **Sessions tab** — shows session history list (read-only, can view past sessions)
- **Progress tab** — shows breakthroughs and insights generated by session-end processing (read-only display)
- **Goals tab** — shows goals from onboarding (read-only display, no editing in v1)
- **Settings tab** — shows account info, sign-out works, other settings are visual only

### Deferred to later versions

- Goal editing UI
- Settings beyond account info and sign-out
- Voice input/output
- Push notifications
- Stripe / subscriptions
- Advanced progress visualizations and metrics
- Native app wrappers

## Progressive Roadmap

- **v1:** Tier 1 + Tier 2 as defined above. Ships to Android testers.
- **v1.x:** Polish and bug fixes from tester feedback.
- **v2:** Goal editing UI, expanded Settings, voice input.
- **v3:** Advanced progress widgets, metrics dashboard, breakthrough detail views.
- **Later:** Subscriptions, notifications, native app wrappers if needed.

The order can shift based on tester feedback. The principle: each version adds working functionality to features that already exist visually, rather than introducing entirely new screens.

## Architecture

v1 uses `reference/prompt-coaching-chat.md` as the session-start system prompt, matching the live Bubble app's API Connector behaviour (literal `Value` field starts with "You are a skilled human c..."). `reference/prompt-v11.3.md` was stored in the Bubble app config (see `app-data-export.json` → `system prompt`) but is **not wired to any live OpenAI call** — it is kept in `reference/` for future prompt iteration work and should not be used in v1. Session-end analysis uses `reference/prompt-session-end-v3.md`.

**No staged pipeline (router + specialists + critic) yet** — that's a deliberate future architecture change after v1 ships and validates with cold users.

Standalone backend with its own database. Not connected to the existing Bubble iOS app. Eventual replacement, not parallel system.

**Database schema designed from day one to accommodate the full data model** — including Goals, Breakthroughs, Insights, Next Steps, Session Feedback, Account Setup, etc. — even if some are read-only or not yet exposed in the UI. This avoids painful migrations later.

## Tech Stack

- **Frontend:** Next.js
- **Hosting:** Vercel
- **Database:** Supabase (Postgres)
- **Auth:** Clerk (magic-link sign-in)
-**AI Model:** OpenAI via `/v1/responses` endpoint

Model usage:
- Session start prompt: gpt-5
- Coaching chat (per-message): gpt-5.2
- Session end analysis: gpt-5

These are deliberate choices in the current Bubble app and must be preserved
for coaching parity. Do not substitute or upgrade without explicit decision.
- **Error tracking:** Sentry
- **Email:** Handled by Clerk for auth emails. No separate transactional email service needed for v1.

## Visual Direction

Match the existing Bubble app and Figma designs. Dark theme with teal and yellow accents. Mobile-first design (most testers will use phones). PWA configured so users can save to home screen.

Reference materials:

## Build Philosophy

- Plan before coding. Approve plans before any meaningful code is written.
- Build in small chunks that can be tested in 5 minutes.
- Boring, simple code. Standard approaches. No clever solutions.
- Don't add features that weren't explicitly asked for.
- Use managed services for anything risky (auth, database, hosting).
- Use git from day one. Commit after every working chunk.
- Ask "what could go wrong" routinely after each chunk.
- Use a `CLAUDE.md` file in the project root for persistent context across sessions.
